  <h2 id="introduction">ВВЕДЕНИЕ</h2>
  <p>REST API широко применяются в современных веб- и мобильных приложениях как основной механизм взаимодействия клиента и сервера. Однако корректная работа API при штатных нагрузках не гарантирует его устойчивости в условиях пиковых запросов, длительной эксплуатации или обработки больших объёмов данных.</p>
  <p>Тестирование производительности представляет собой комплексную методологию верификации нефункциональных характеристик системы. В отличие от функционального тестирования, оно направлено не на корректность логики, а на её устойчивость в условиях, приближенных к реальной эксплуатации: высокая конкуренция запросов, длительная работа, обработка больших массивов данных.</p>
  <p>В контексте REST API выделяют три основных вида производительного тестирования:</p>
  <ol>
    <li><strong>Нагрузочное</strong> — для определения границ нормальной работы;</li>
    <li><strong>Стрессовое</strong> — для выявления пределов отказоустойчивости и поведения в аварийных сценариях;</li>
    <li><strong>Объемное</strong> — для оценки стабильности при работе с большими объёмами данных.</li>
  </ol>
  <p>Объективная оценка результатов невозможна без измерения ключевых метрик: времени отклика, пропускной способности, уровня ошибок и др.</p>
  <p><strong>Цель реферата</strong> — изучить и охарактеризовать указанные виды тестирования и систематизировать основные метрики производительности REST API.</p>

  <h2 id="section1">1. ОБЩИЕ СВЕДЕНИЯ О REST API И ЕГО ПРОИЗВОДИТЕЛЬНОСТИ</h2>

  <h3 id="1.1">1.1 Краткая характеристика REST-архитектуры</h3>
  <p>REST (Representational State Transfer) — это стиль архитектуры программного обеспечения для распределенных систем, таких как World Wide Web, который, как правило, используется для построения веб-служб. Термин REST был введен в 2000 году Роем Филдингом, одним из авторов HTTP-протокола. Системы, поддерживающие REST, называются <em>RESTful</em>-системами [2].</p>
  <p>В общем случае REST является очень простым интерфейсом управления информацией без использования каких-то дополнительных внутренних прослоек. Каждая единица информации однозначно определяется глобальным идентификатором, таким как URL. Каждая URL в свою очередь имеет строго заданный формат [2].</p>
  <p>Как происходит управление информацией сервиса — это целиком и полностью основывается на протоколе передачи данных. Наиболее распространенный протокол — <strong>HTTP</strong>. Для HTTP действие над данными задаётся с помощью методов:</p>
  <ol>
    <li><strong>GET</strong> — получение данных с сервера;</li>
    <li><strong>POST</strong> — создание нового ресурса на сервере или отправка данных для обработки;</li>
    <li><strong>PUT</strong> — обновление существующего ресурса или создание ресурса, если он не существует;</li>
    <li><strong>DELETE</strong> — удаление ресурса с сервера.</li>
  </ol>
  <p>Таким образом, действия CRUD (Create-Read-Update-Delete) могут выполняться как со всеми 4-мя методами, так и только с помощью GET и POST.</p>

  <h3 id="1.2">1.2 Факторы, влияющие на производительность API</h3>
  <p>Интерфейс прикладного программирования (API) — это сложный механизм, который обеспечивает беспрепятственное взаимодействие и связь между различными программными системами путем создания общего языка и набора протоколов. API позволяют разработчикам использовать готовые компоненты и сервисы, тем самым ускоряя процесс разработки программного обеспечения и способствуя модульности [5].</p>
  <p>Производительность API, важный аспект программной инженерии, относится к эффективности и отзывчивости интерфейса прикладного программирования, когда он подвергается ряду функциональных требований. Этот критерий включает в себя такие аспекты, как задержка, пропускная способность и надежность, которые в совокупности определяют пользовательский опыт и общую эффективность системы.</p>
  <p>Проблема производительности API заключается в достижении тонкого баланса между быстрым и надежным предоставлением услуг и соблюдением ограничений инфраструктуры, бюджета и масштабируемости.</p>
  <p>На производительность API влияют факторы, связанные с состоянием сервера, параметрами сети, эффективностью работы базы данных и качеством кода:</p>
  <ol>
  <li><strong>Сервер</strong>:
    <ul>
      <li><em>Время обработки запроса</em>. Сервер тратит время на обработку запроса, выполнение необходимых операций и возврат ответа. На это влияют оборудование сервера, эффективность базы данных и сложность логики API. Оптимизировать время обработки запроса сервером можно путем модернизации аппаратного обеспечения сервера, кэширования часто используемых данных и упрощения логики API [3].</li>
      <li><em>Перегрузка сервера</em>. Например, запросы, требующие много ресурсов, или запросы в периоды высокого трафика, которые создают дополнительную нагрузку на сервер. В таких случаях важно заранее подготовить стратегию для обработки пиковых нагрузок, которая временно отключает запросы к перегруженному сервису.</li>
    </ul>
  </li>
  <li><strong>Сеть</strong>:
    <ul>
      <li><em>Задержка сети</em>. На задержку влияют географическое расстояние между клиентом и сервером, перегрузка сети и качество интернет-соединения [3].</li>
      <li><em>Размер полезной нагрузки</em>. Большие данные дольше передаются и обрабатываются, что увеличивает время отклика [3].</li>
      <li><em>Использование современных сетевых протоколов</em>.</li>
    </ul>
  </li>
  <li><strong>База данных</strong>:
    <ul>
      <li><em>Оптимизация запросов к базе данных</em>. Важно правильно индексировать данные, удалять избыточные операции и выбирать только необходимые поля.</li>
      <li><em>Использование пула соединений</em>. Вместо создания нового соединения с базой данных для каждого запроса, что может замедлить работу из-за времени, необходимого на установку соединения, пул соединений позволяет заранее создать набор активных соединений, которые могут быть повторно использованы различными запросами.</li>
      <li><em>Пагинация</em>. Если API возвращает большое количество записей, можно ограничить количество записей в каждом запросе, тем самым уменьшая объём передаваемых данных по сети.</li>
    </ul>
  </li>
  <li><strong>Код</strong>:
    <ul>
      <li><em>Оптимизация алгоритмов</em>. Направлена на повышение эффективности выполнения операций, заложенных в коде [3].</li>
      <li><em>Использование асинхронных методов работы</em>. Позволяет API обрабатывать несколько запросов одновременно, не ожидая завершения длительных операций [3].</li>
    </ul>
  </li>
</ol>

  <h2 id="section2">2. ВИДЫ ТЕСТИРОВАНИЯ ПРОИЗВОДИТЕЛЬНОСТИ REST API</h2>
  <p>Тестирование производительности включает в себя измерение различных метрик с целью проверить производительность системы при различных уровнях трафика и пропускной способности.</p>

  <h3 id="2.1">2.1 Нагрузочное тестирование</h3>
  <p>Нагрузочное тестирование оценивает поведение компонента или системы в условиях увеличивающейся нагрузки, например при увеличении количества параллельных пользователей и/или количества транзакций, чтобы понять, выдержит ли этот компонент или система подобную нагрузку.</p>
  <p>Упор делается на ожидаемую и реалистичную нагрузку, хотя в основу этого вида тестирования включают разнообразные сочетания запросов и их количество. Запросы создаются таким образом, чтобы смоделировать одновременную работу набора пользователей. Это позволяет оценить время ответа и пропускную способность [1].</p>

  <h3 id="2.2">2.2 Стрессовое тестирование</h3>
  <p>Это разновидность тестирования с целью изучения поведения системы или компонента при пиковых объёмах нагрузки и в необычных условиях функционирования, например при нехватке таких ресурсов, как память или доступ к серверам. Стрессовое тестирование — это выкрученное на максимум нагрузочное тестирование. Его цель — убедиться, что время ответа, надёжность и функциональность будут деградировать медленно и предсказуемо. Не должно быть асоциального поведения со стороны системы: повреждения данных, блокировки системы или её падения [1].</p>

  <h3 id="2.3">2.3 Объемное тестирование</h3>
  <p>Объёмное тестирование — это вид нефункционального тестирования производительности, направленный на оценку поведения системы при обработке больших объёмов данных, а не большого количества одновременных пользователей или запросов. В отличие от нагрузочного или стрессового тестирования, фокус здесь делается на размере данных, передаваемых через интерфейс или хранящихся/обрабатываемых в системе.</p>
  <p>Основными целями объёмного тестирования являются:</p>
  <ol>
    <li>Проверка стабильности и корректности работы API при передаче или обработке «тяжёлых» полезных нагрузок.</li>
    <li>Выявление проблем с памятью: утечки, неоптимальное использование буферов, превышение лимитов.</li>
    <li>Оценка масштабируемости хранилищ и СУБД: рост времени выполнения запросов с увеличением объёма таблиц, фрагментация индексов, деградация скорости записи/чтения.</li>
    <li>Проверка корректности обработки данных при изменении объёма: сохраняются ли ограничения валидации, не возникают ли ошибки форматирования или потери данных.</li>
  </ol>

  <h2 id="section3">3. Ключевые метрики производительности REST API</h2>
  <p>Показатели производительности API, важнейший компонент программной инженерии, включают в себя целый ряд количественных показателей, которые позволяют разработчикам тщательно оценить и оптимизировать эффективность и быстродействие интерфейсов прикладного программирования [2].</p>
  <p>Выделяют следующие метрики:</p>
  <ol>
    <li><strong>Время отклика</strong> — интервал между отправкой клиентом запроса и получением полного ответа. Часто синонимизируется с латентностью, хотя, строго говоря, латентность — это минимально возможная задержка в системе (включая сетевую), тогда как время отклика включает также время обработки на сервере.</li>
    <li><strong>Пропускная способность</strong> — количество запросов или транзакций, успешно обработанных системой в единицу времени. В отличие от латентности, пропускная способность отражает масштабируемость и общую производительность системы под нагрузкой. Высокая пропускная способность при низкой латентности свидетельствует об эффективной архитектуре и оптимизированной логике обработки.</li>
    <li><strong>Уровень ошибок</strong> — доля неуспешных запросов относительно общего числа запросов за фиксированный период. Повышенный уровень ошибок, даже при нормальных латентности и пропускной способности, указывает на функциональные или инфраструктурные проблемы: истечение таймаутов, исчерпание ресурсов, ошибки валидации или сбои зависимостей.</li>
    <li><strong>Метрики потребления ресурсов</strong> (процессорное время, объём используемой памяти, дисковые и сетевые операции ввода-вывода) дают представление о внутреннем состоянии системы и помогают выявлять узкие места. Особое внимание уделяется ресурсам СУБД, так как большинство REST API в значительной степени зависят от скорости выполнения SQL-запросов.</li>
    <li><strong>Время до первого байта (TTFB)</strong> — время от момента получения запроса сервером до отправки первого байта ответа клиенту. Эта метрика отражает скорость инициализации обработки и включает: маршрутизацию, логику контроллера и, зачастую, выполнение первого SQL-запроса.</li>
    <li><strong>Перцентили</strong> — это показатели, которые используются для описания распределения данных, избегая искажений, характерных для среднего значения. Например, если время отклика сервиса составляет p95 = 100 мс, это значит, что в 95% случаев отклик занимает 100 мс или меньше, а в 5% случаев может превышать это значение. В высоконагруженных системах мониторинг p99 критичен для обеспечения предсказуемости поведения даже в пиковые моменты нагрузки [4].</li>
  </ol>
  <p>Эти метрики обычно анализируются в совокупности: например, рост пропускной способности с одновременным увеличением p99 и уровнем ошибок может свидетельствовать о приближении к пределу ёмкости системы.</p>

  <h2 id="conclusion">ЗАКЛЮЧЕНИЕ</h2>
  <p>Проведённый анализ показал, что тестирование производительности REST API является неотъемлемой частью обеспечения качества современных программных систем. Нагрузочное, стрессовое и объёмное тестирование, несмотря на общую цель — верификацию устойчивости API, — решают принципиально разные задачи и требуют индивидуального подхода к проектированию сценариев и интерпретации результатов.</p>
  <p>Эффективность оценки напрямую зависит от выбора и комплексного анализа ключевых метрик: времени отклика, пропускной способности, уровня ошибок, времени до первого байта, перцентилей и показателей использования ресурсов. Их совместное рассмотрение позволяет не только зафиксировать факт деградации, но и точно локализовать её причину: неоптимизированный запрос к БД, утечка памяти или исчерпание соединений.</p>
  <p>Таким образом, систематическое применение методов производительного тестирования способствует повышению надёжности, масштабируемости и предсказуемости API, что в конечном итоге обеспечивает соответствие требованиям пользователей и бизнес-показателям.</p>
